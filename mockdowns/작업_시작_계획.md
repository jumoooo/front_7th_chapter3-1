# Chapter3-1 작업 실행 로드맵

> 🗂️ **문서 목적:** 과제 요구사항(Atomic Design 재해석, shadcn/ui, TailwindCSS 4.x, CVA, Storybook)을 순서대로 수행하기 위한 구체적인 실행 계획을 정의한다.

---

## 1. 준비 상태 점검

- ✅ TailwindCSS 4.x + shadcn/ui 기본 셋업 완료 (`packages/after` 기준)
- ✅ `mockdowns/셋업_완료_문서.md`에 환경 구축 기록 존재
- ⛳ **미완료 핵심 과제**
  - After 패키지 디자인 시스템 구축
  - Before → After 마이그레이션
  - Storybook 스토리 작성
  - README 문서화 (Before/After 비교)

---

## 2. 전체 워크플로우 개요

1. **Before 패키지 분석 및 문제점 기록**
2. **디자인 토큰 & 공통 컴포넌트 정리 (TailwindCSS + CVA + shadcn/ui)**
3. **도메인 컴포넌트/페이지 마이그레이션 (ManagementPage)**
4. **Storybook 구성 및 컴포넌트 문서화**
5. **README 및 회고 문서 업데이트**

각 단계를 완료할 때마다 mockdowns 문서에 체크리스트 형태로 기록한다.

---

## 3. 단계별 상세 계획

### 3.1 Before 패키지 분석
- 📍 경로: `packages/before`
- 🛠️ 작업
  - 프로젝트 실행 (`pnpm dev:before`) 후 화면 캡처 및 동작 확인
  - 컴포넌트별 문제점(스타일, 설계, 타입, 접근성) 기록
  - `mockdowns/실무_진행_계획.md`에 “Before 분석” 섹션 업데이트
- ✅ 산출물: 문제 리스트 + 개선 필요 이유

### 3.2 디자인 시스템 기반 구축 (After 패키지)
- 📍 경로: `packages/after`
- 🛠️ 작업
  - `components/ui/`에 shadcn/ui 컴포넌트 활용
  - 기존 `atoms/molecules/organisms` 컴포넌트는 UI 순수화를 위해 CVA 적용
  - TailwindCSS 토큰(`src/styles/globals.css`) 보완
  - 도메인/비즈니스 로직은 상위 레벨에서 처리하도록 분리
- ✅ 산출물: 순수 UI 컴포넌트 세트 + variants 패턴

### 3.3 ManagementPage 마이그레이션
- 📍 경로: `packages/after/src/pages/ManagementPage.tsx`
- 🛠️ 작업
  - Before 구조 참고하여 shadcn/ui + 리팩토링된 컴포넌트로 재구성
  - 상태 관리 / 서비스 호출 로직은 hooks 혹은 services로 정리
  - Modal, Table, Form 등 UI → `components/ui/` 기반 사용
- ✅ 산출물: After 패키지에서 동일 기능 동작

### 3.4 Storybook 구성
- 📍 경로: `packages/after`
- 🛠️ 작업
  - Storybook 설치 (`pnpm dlx storybook@latest init`)
  - 주요 컴포넌트(Button, Input, Select, Card, Table, Form) stories 작성
  - 접근성(Addon) 및 Controls 설정
- ✅ 산출물: `stories` 디렉토리 + 실행 가능한 Storybook

### 3.5 문서화 및 제출 준비
- 📍 경로: 루트 README, mockdowns 폴더
- 🛠️ 작업
  - README에 Before/After 비교, 개선 설명, 실행 방법 정리
  - `mockdowns/셋업_완료_문서.md`와 `실무_진행_계획.md` 업데이트
  - 과제 회고(문제점/집중한 부분/질문) 작성
- ✅ 산출물: README + mockdowns 문서 최신화

## 5. 작업 시 유의사항

- 🎯 디자인 시스템 폴더 구조는 **역할 기반**으로 단순화 (`components/ui/`, `components/domain/` 등)
- ♻️ UI 컴포넌트는 **비즈니스 로직 분리**에 집중
- 🧪 모든 변경은 Storybook / Vitest로 가능하면 검증
- 📚 변경 후 반드시 mockdowns 문서 업데이트
